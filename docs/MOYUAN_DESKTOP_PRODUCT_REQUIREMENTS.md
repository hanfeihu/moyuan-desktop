# 墨渊 Desktop 产品需求文档

## 1. 产品定位

墨渊 Desktop 是面向企业员工的本地化 AI 工作台。它的核心不是做一个普通聊天机器人，而是把 Codex 类能力内置到企业员工日常工作环境里，让员工可以直接让 AI 读项目、执行命令、修改文件、生成内容、整理日报，并把过程沉淀为企业可控、可审计、可恢复的工作上下文。

产品目标可以概括为：

- 对员工：开箱即用的 AI 同事，安装客户端后不需要单独安装 Codex，就能在本地工作区完成编码、文档、分析、图片生成等任务。
- 对企业：员工数据、组织权限、模型渠道、审计策略、绩效数据沉淀均可控，支持私有化/本地化部署。
- 对管理者：第一阶段以日报、任务过程、工作成果沉淀和绩效辅助评估为切入点，但采集范围必须透明、可配置、可审计。

品牌名称：墨渊 / 墨渊 AI / 墨渊 Desktop。可使用现有域名 `moyuanai.com` 承接官网、后台入口、企业部署文档和下载页。

## 2. 产品形态

### 2.1 Desktop 员工端

Desktop 是员工每天使用的主产品，优先级最高。

核心能力：

- 本地安装后即可使用，不要求员工再安装 Codex。
- 内置 Codex 原生任务执行体验，目标对齐 `exec`、`resume`、`fork`、本地命令执行、文件读写、diff、会话恢复等能力。
- 可连接本地工作区，围绕当前项目形成上下文记忆。
- 支持多轮对话续聊，历史会话可恢复，打开历史对话时默认滚动到底部。
- 支持流式响应，让员工明显感觉到 AI 正在思考和执行，而不是等待一次性结果。
- 支持图片生成作为隐式工具能力：员工只需要自然输入“生成一张图/做一个海报/画一个封面”，系统自动调用图片能力并把结果插入对话。
- 支持企业自定义能力入口，例如日报、绩效、知识库、审批、组织通讯录等，但这些入口在 UI 上必须克制，不能破坏 Codex 式工作台气质。

### 2.2 Admin 后台

Admin 是企业管理员、HR、IT、部门负责人使用的 Web 后台。建议使用 Ant Design Pro 或同等级企业后台框架，保证列表、权限、审计、配置、报表体验成熟稳定。

核心能力：

- 企业、部门、员工、角色、权限管理。
- 企业微信、飞书、钉钉组织架构数据源配置。
- 模型网关、中转地址、密钥、模型、额度、策略配置。
- Desktop 客户端版本、更新、准入策略管理。
- 工作区策略、命令执行策略、文件访问策略、敏感词和脱敏策略。
- 日报、任务记录、成果沉淀、绩效辅助分析。
- 审计日志、会话检索、异常行为告警。
- 插件/工具能力管理，例如图片生成、知识库检索、企业应用调用。

## 3. 组织数据源

企业员工信息和组织架构以企业微信、飞书、钉钉为主要来源。

必须支持：

- 企业级授权接入。
- 部门、成员、角色、直属上级、岗位、在职状态同步。
- 增量同步和定时全量校准。
- 离职/转岗/部门变更后的权限自动更新。
- 本地账号与外部组织账号映射。
- 多租户隔离，企业之间数据完全隔离。

第一阶段建议接入顺序：

1. 企业微信。
2. 飞书。
3. 钉钉。

原因：企业微信和飞书在目标企业办公场景里更常见，且组织架构、消息、审批、日报等生态更适合作为切入点。

## 4. 本地化部署

墨渊需要支持企业本地化/私有化部署，优先使用 Docker 化部署方式。

部署内容：

- Admin 后台。
- API 服务。
- Codex Runtime 服务。
- 数据库服务。
- 文件/图片/附件存储。
- 模型中转配置。
- 日志与审计服务。

部署要求：

- 所有密钥通过环境变量或后台安全配置注入，不写入代码仓库和文档。
- 数据库、对象存储、日志存储可替换，适配企业已有基础设施。
- 支持离线或半离线企业环境下的升级包交付。
- Desktop 客户端可以连接企业内网服务。
- 企业可配置是否允许外部模型服务、图片生成服务、插件市场、远程工具调用。

## 5. Codex 原生能力要求

Desktop 的第一目标是实现 Codex 类桌面工作体验，而不是普通 Web Chat。

必须具备：

- 任务执行：用户提交任务后，系统能够调用 Codex 原生执行能力处理本地工作区任务。
- 本地命令：具备调用本地命令行的能力，但必须受企业策略、用户确认和审计约束。
- 文件理解：能读取项目文件、理解目录结构、定位问题。
- 文件修改：能修改文件，并展示变更结果。
- diff 展示：文件变更需要统一进入上下文，可在 UI 中查看。
- 命令历史：执行过的命令和结果应纳入任务上下文。
- 运行日志：任务执行过程中的关键状态需要可见，但不要把底层噪声暴露给普通员工。
- 错误恢复：任务失败时要给出可理解原因，并允许继续修复。
- 无需员工单独安装 Codex：客户端应内置或随包携带运行所需能力。

体验标准：

- 员工输入自然语言后，墨渊像一个真正的工作代理一样行动。
- 不让员工理解底层 `exec`、`resume`、runtime、端口等技术概念。
- UI 上呈现的是“正在做什么、做到了哪里、结果是什么、下一步怎么继续”。

## 6. 会话持久化、Resume、Fork 与上下文

墨渊必须不是“一次输入，一次任务执行”的短会话工具，而是可长期工作的上下文系统。

必须支持：

- 多轮对话续聊。
- 每个任务保存完整 transcript。
- 下一轮任务自动携带上一轮 transcript 中的关键上下文。
- 会话持久化到本地数据库。
- 支持恢复历史会话，类似 Codex resume。
- 支持从历史节点派生新分支，类似 fork。
- 工作区级上下文记忆，包括项目路径、常用命令、技术栈、用户偏好、企业规则。
- 文件变更 diff、命令历史、任务状态统一进入上下文。
- 历史会话打开后默认滚动到底部，输入框固定在底部，不丢失草稿。

上下文分层：

- 当前消息：用户本轮明确要求。
- 当前任务 transcript：本次任务完整过程。
- 会话记忆：同一对话下的历史目标、决策和结果。
- 工作区记忆：同一项目下长期稳定的信息。
- 企业记忆：组织规则、合规要求、可用工具、统一提示词。

## 7. 企业可控与审计

企业可控是墨渊与个人 AI 工具的核心区别。

必须支持：

- 企业管理员配置模型渠道、模型权限、额度和审计级别。
- 管理员配置员工可访问的工作区、命令、文件范围。
- 高风险命令需要确认、拦截或审批。
- 任务、命令、文件变更、模型调用、工具调用、图片生成均进入审计日志。
- 敏感信息可脱敏、屏蔽或禁止发送到外部模型。
- 日报、任务成果、代码变更、文档产出可形成绩效辅助数据。
- 员工端需要明确展示企业策略，不做隐蔽采集。

绩效切入点建议：

- 第一阶段不要做“监控员工”的产品表达，而是做“自动沉淀日报和成果”的企业效率工具。
- 日报从任务完成记录、提交记录、会话摘要、文件变更、手动补充中生成。
- 管理端提供部门/个人工作成果视图，辅助绩效沟通，但最终评价仍由管理者确认。

## 8. 图片生成作为隐式工具能力

图片生成不是一个单独模式，也不是让用户切到“图片应用”。它应该是墨渊大脑可调用的工具能力。

产品原则：

- 不在默认输入框暴露“图片模式”切换。
- 用户自然输入图片需求时，系统自动判断并调用图片生成工具。
- 图片生成能力写入基础提示词/能力说明，让模型知道自己拥有该工具。
- 对用户表现为同一条对话流里的自然回复。
- 生成过程中显示类似“正在生成图片...”的状态。
- 完成后在对话里直接渲染图片，可继续让用户修改、重画、换风格。

能力边界：

- 图片生成接口、模型、额度、密钥由后台配置。
- 密钥不得出现在客户端代码、文档、日志和前端可见配置里。
- 若用户请求视频生成但当前只接入图片生成，应明确提示当前可先生成关键帧/封面图，视频能力后续接入。
- 若用户是在讨论“图片生成功能代码”，不应误触发图片生成。

## 9. 基础提示词与工具说明

每次提交任务时，墨渊应给底层模型注入企业和能力上下文，让模型知道它能做什么、不能做什么。

基础提示词应包含：

- 你是墨渊 Desktop，企业员工的本地 AI 工作代理。
- 你可以读取和理解当前工作区文件。
- 你可以在受控策略下执行本地命令。
- 你可以修改文件并返回 diff。
- 你可以继续历史会话，必要时引用上一轮 transcript。
- 你可以使用图片生成工具，但该能力是隐式工具，不要求用户切换模式。
- 你必须遵守企业策略、数据安全、审计和最小权限原则。
- 当任务涉及高风险命令、敏感数据或不可逆操作时，需要提示用户确认或按企业策略拦截。
- 输出要面向员工，尽量使用自然、清晰、非技术负担过重的语言。

提示词不应包含：

- 任何 API key、token、密钥。
- 具体不可公开的内部服务凭证。
- 与当前企业无关的测试账号。

## 10. UI 与交互标准

整体气质：接近 Codex Desktop，克制、优雅、清爽、有质感。不能像粗糙后台，也不能像普通聊天网页。

视觉标准：

- 字体优先使用 Geist 与 Noto Sans SC，中文、英文、数字混排要自然。
- 色调以温和浅色为主，减少突兀高饱和色。
- 控件留白要舒服，但信息密度不能过低。
- 滚动条要弱化或隐藏，不能出现突兀的系统默认滚动条。
- 左上角品牌、窗口区域、历史列表、底部企业入口都要精致，不要像临时 demo。

输入框标准：

- 输入框固定在页面底部。
- 结构应接近 Codex composer：大圆角白色输入卡片，上方可有上下文/草稿条，中间是输入区，底部是工具栏。
- 不使用显式“图片模式”按钮。
- 左侧可保留 `+`、自定义能力、企业入口等。
- 右侧可显示模型、策略、发送/停止状态。
- 发送后输入框不消失，不影响页面上下滚动。
- 支持多行输入，高度增长要自然，不能遮挡消息。

对话流标准：

- AI 回复必须流式展示。
- 任务状态要有“正在思考、正在执行、正在生成图片、正在修改文件”等过程感。
- 用户消息、AI 消息、工具结果、图片结果、diff 结果有清晰层级。
- 点击历史对话后默认定位到底部。
- 新消息到达时，如果用户正在底部，自动跟随到底部；如果用户在看历史，则不强行打断。
- 错误要以人能理解的方式展示，并给出继续处理入口。

桌面体验标准：

- 窗口可拖动。
- 标题栏和左上角区域不能粗糙。
- 刷新、重启、服务异常恢复尽量由系统处理，不让用户手动处理。
- 客户端启动时应检查 runtime 状态，必要时自动启动或提示一键修复。

## 11. 今晚交付范围

今晚目标是做成可体验的产品雏形，而不是残次 demo。

Desktop 必须交付：

- 可启动的桌面客户端。
- Codex 类输入、任务执行、流式回复体验。
- 本地工作区任务执行能力。
- 多轮会话续聊。
- 历史会话恢复，打开后滚动到底部。
- 会话持久化。
- 工作区记忆基础能力。
- 文件变更 diff 和命令历史进入上下文。
- 图片生成作为隐式工具能力。
- 底部固定 Codex 式 composer。
- 左侧历史与企业入口基础框架。
- 基础异常恢复和错误提示。

Admin 可先交付基础框架：

- Ant Design Pro 风格后台骨架。
- 企业、员工、组织、模型配置、审计、日报/绩效模块入口。
- 真实数据接入可作为后续迭代，但信息架构今晚需要定下来。

文档必须交付：

- 产品需求文档。
- 验收测试清单。
- 风险与后续路线。

## 12. 验收测试清单

### 12.1 Desktop 基础流程

- 启动 Desktop 后不需要用户手动打开 runtime。
- 输入普通任务，系统能正常回复。
- 回复是流式展示，不是等待后一次性出现。
- 输入框始终固定在底部。
- 页面可上下滚动，输入框不丢失。
- 发送后按钮状态从发送切换为运行/停止态。
- 任务失败时显示可理解错误，并允许继续。

### 12.2 Codex 能力

- 选择一个本地工作区后，系统能理解项目结构。
- 用户要求查看文件时，能读取对应文件。
- 用户要求修改代码时，能产生文件变更。
- 文件变更能被记录并进入上下文。
- 命令执行记录能被保存。
- 下一轮对话能引用上一轮结果。
- 重启应用后可以恢复历史会话。
- 从历史会话继续提问时不会变成全新任务。

### 12.3 会话体验

- 左侧点击历史会话后，右侧默认滚动到底部。
- 历史会话消息、工具过程、图片结果都能正确显示。
- 用户输入草稿不会因为切换会话意外丢失。
- 长对话滚动条不突兀。
- 新消息到达时，底部跟随逻辑自然。

### 12.4 图片生成

- 输入“帮我生成一张红苹果在白色桌子上的图”，系统自动调用图片能力。
- UI 不出现“图片模式”切换。
- 生成过程显示正在生成状态。
- 生成成功后图片直接显示在对话中。
- 用户继续说“换成蓝色背景”，系统能理解是在修改上一张图的意图。
- 输入“帮我改一下图片生成功能代码”时，不应误触发图片生成。
- 未配置图片接口时，系统给出清晰提示，不泄露内部配置。

### 12.5 企业能力

- Admin 能看到组织、员工、权限、模型配置、审计、日报/绩效入口。
- Desktop 能体现企业身份和企业策略存在。
- 高风险命令不会静默执行。
- 模型中转配置不出现在前端可见代码里。
- 审计日志不记录明文密钥。

### 12.6 UI 体验

- 字体、字号、行高在中文场景下自然。
- composer 不遮挡内容。
- 左上角品牌区域精致。
- 左侧企业入口固定在底部，按钮数量不被随意删改。
- 整体不像后台模板，也不像普通聊天页面。
- 视觉质感接近 Codex Desktop：克制、轻、稳、有过程感。

## 13. 风险与后续

### 13.1 风险

- Codex 相关能力的许可证、分发方式、内置方式需要确认，避免商业交付风险。
- 本地命令执行具有安全风险，需要企业策略、确认机制、审计和最小权限。
- 员工数据采集涉及隐私和劳动合规，必须透明告知、可配置、可审计。
- 图片生成接口如果走外部服务，需要明确数据出境、内容安全、额度和失败兜底。
- 长会话上下文容易膨胀，需要摘要、压缩、分层记忆机制。
- UI 若只追求“像 Codex”但忽略企业功能，会影响后续扩展；需要在克制设计中保留企业入口。

### 13.2 后续路线

- 完成企业微信组织架构接入。
- 接入飞书与钉钉。
- 完善 Admin 权限和审计中心。
- 增加日报自动生成、确认、提交、主管查看流程。
- 增加团队知识库和企业应用工具调用。
- 增加更完整的 fork 会话分支视图。
- 增加模型/工具调用成本统计。
- 增加客户端自动更新和企业灰度发布。
- 增加图片编辑、参考图、风格模板和企业素材库。
- 增加安全策略模板，例如研发、财务、人事、销售不同角色策略。

## 14. 当前优先级判断

最高优先级是把 Desktop 做到“员工愿意每天打开”的程度。

当前最关键的产品判断：

1. Desktop 先于 Admin。
2. Codex 原生体验先于普通聊天体验。
3. 流式、续聊、上下文、恢复、diff、命令历史是基础，不是加分项。
4. 图片生成是隐式能力，不是显式模式。
5. 企业可控要从第一天进入架构，而不是后补。
6. UI 必须按产品级标准打磨，不能停留在技术 demo。
